報告書：Genesys Logic Firmware 安全簽署與驗證流程 (Code Signing)

## 💡 簡介 (Introduction)

Genesys Logic 為了滿足 HP Code Signing 的安全要求，設計了一套完整的韌體簽署與驗證架構。此架構的核心目標是確保所有簽署操作都能符合 FIPS 140-2 Level 3 的安全標準，並採用 ECDSA nistp256 加密演算法。在滿足嚴格安全合規性的前提下，Genesys Logic 還需要建立一套兼具成本效益、可擴展性與災難恢復能力的內部簽署系統。

## 🔑 Code Signing 資源與實作方案 (含 FIPS 合規要求)

Genesys Logic 選擇採用「經 FIPS L3 認證的 USB eToken 硬體，並搭配自行開發的 SDK 應用程式」作為最終方案 [`3rd-party-code-signing-specification-ecdsa`]。此方案將金鑰安全地儲存在防竄改硬體中，同時透過自研的金鑰管理系統（KMS）實現高度客製化的簽署流程與嚴格的稽核管控。

## 🏛️ eToken 簽署系統架構解析

Genesys Logic 的 eToken 簽署系統架構包含以下主要元件：

1. FIPS L3 認證的 USB eToken 硬體設備，用於安全地儲存私密金鑰。
2. 自行開發的 SDK 應用程式，提供金鑰管理、檔案簽署等功能。
3. 金鑰管理系統（KMS），負責金鑰的生成、佈署和管理。

此架構確保私密金鑰始終安全地保存在防竄改的硬體設備中，並透過自研的 SDK 和 KMS 實現高度客製化的簽署流程和嚴格的稽核管控。

## 🔑 Genkey 金鑰生成與佈署流程解析

Genesys Logic 的金鑰生成與佈署流程如下：

1. 在 KMS 中生成 ECDSA nistp256 金鑰對。
2. 將公鑰部署到簽署系統的信任存儲中。
3. 將私鑰安全地儲存在 FIPS L3 認證的 eToken 硬體設備中。

此流程確保私鑰始終受到嚴格的安全保護，並透過 KMS 實現金鑰的集中管理和自動化佈署。

## 📜 Sign File 簽署檔案工作流程

Genesys Logic 的檔案簽署工作流程如下：

1. 使用者將待簽署的檔案傳送至簽署系統。
2. 簽署系統讀取 eToken 中的私鑰，並使用 ECDSA nistp256 演算法對檔案進行數位簽章。
3. 簽署系統將簽章資訊附加到原始檔案，生成最終的已簽署檔案。

此流程確保檔案簽署過程完全符合 FIPS 140-2 Level 3 的安全標準。

## 🔄 端到端的程式碼簽署與驗證完整流程

Genesys Logic 的端到端程式碼簽署與驗證流程如下：

1. 檔案簽署：使用 eToken 中的私鑰對檔案進行數位簽章。
2. 檔案傳輸：將已簽署的檔案傳輸至目標設備。
3. 檔案驗證：目標設備使用預先部署的公鑰驗證檔案的數位簽章，確認其合法性。

此流程確保整個程式碼生命週期都能得到有效的安全保護。

## 🚀 程式碼簽署與驗證流程 (Boot Code Verify)

Genesys Logic 的啟動程式碼簽署與驗證流程如下：

1. 啟動程式碼簽署：使用 eToken 中的私鑰對啟動程式碼進行數位簽章。
2. 啟動程式碼傳輸：將已簽署的啟動程式碼傳輸至目標設備。
3. 啟動程式碼驗證：目標設備使用預先部署的公鑰驗證啟動程式碼的數位簽章，確認其合法性。
4. 啟動程式碼執行：只有在驗證通過的情況下，目標設備才會執行啟動程式碼。

此流程確保啟動程式碼的完整性和合法性，防止未經授權的程式碼執行。

## eToken 集中式管理與安全存取服務平台(未來架構)

為了進一步提升 Genesys Logic 的程式碼簽署系統的可擴展性和管理效率，未來可以考慮建立一個集中式的 eToken 管理與安全存取服務平台。此平台將提供以下功能：

1. 集中管理多個 eToken 設備，實現金鑰和簽署操作的統一管理。
2. 透過身份驗證和訪問控制機制，確保只有經授權的用戶和系統才能存取 eToken 設備。
3. 提供遠程簽署服務，使用戶無需直接訪問 eToken 設備即可完成簽署操作。
4. 記錄所有簽署操作的詳細審計日誌，實現對簽署流程的全面監控和追溯。

此未來架構將進一步提升 Genesys Logic 程式碼簽署系統的安全性、可擴展性和管理效率。
