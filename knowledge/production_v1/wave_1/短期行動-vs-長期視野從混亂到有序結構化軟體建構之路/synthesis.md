# 短期行動 vs 長期視野，從混亂到有序，結構化軟體建構之路

## 起點：挑戰與混亂
8年前，我進入公司工作，工作內容與以前有很大差別。以前公司是以專案為主，完全會跟著產品生產的流程來決定軟體要做到哪個地步。但是現在的做法是根據各業務單位 (BU) 的需求來提供服務，所以BU有什麼想法，我們就要做出相應的工具。這造成需求變得多又雜亂，導致為了趕時間，只要求工具能用就好，就變成了我進來時看到的情況：

- 有SVN版控，但並非所有工具都有，只有長期Release的工具才有。
- BU所需的工具根據每個人的習慣不一定。
- 因為BU所需要的工具，最基本的程式碼還是大同小異，為了快速改完需求，直接從SVN裡面複製一份出來改是最快的，就導致每份工具裡面都會看到很多類似的程式碼。

## 第一步的修正
先以底層程式碼統一為主，先把所有目前有用到的原始程式碼的底層做整合，讓所有的工具都呼叫同一個底層的程式碼，但是還是以BU的需求為主，有空閒時再修正相關的程式碼：

- 整理所有用到的工具，分析是否有底層程式碼可以合併的。
- 整理工具相關文件，讓雜亂無章的資訊可以整合起來。

## 第二步的改善
之前的同事基本上都是配合硬體開發軟體，並沒有純軟體出生，或是說對軟體沒有熱烈喜好。Adam 進來後補足了這一塊，在請Adam修改對應功能時，他開發時間會比較久，所以會趁開發時，也會把對應的程式碼做重構 (refactor)。發現這樣的現象後，也開始調整給Adam的工作，讓他可以開發比較長期的需求，並根據自己調整重構的比例：

- 根據客戶的需求分析工具的使用率。
- 將最常用的工具先做重構。
- 重構的程式碼和初始的程式碼作為停損點，在指定時間內把重構的程式碼導入工具裡面。
- 重點工具都有讓成員寫相關說明文件，確保不同人來做時可以更快上手。

## 第三步的大躍進
因為BU的需求越來越多，導致人力不足，這時人從3人 -> 6人，人數變多，讓調配工作的彈性更高，讓某些對原始程式碼有潔癖的人做長期開發的工作，讓靈活的人做溝通和短期開發，BU短期的需求可以準時交出來。原始程式碼的重構程度越高，短期需求所花的力氣就更少，讓我們可以專注在更想要做的事情上面，也更能做長期的規劃，讓客戶的需求更容易辦到：

- 從 SVN 遷移到 GitHub。
- 導入程式碼審查 (code review) 機制。
- 重點專案導入 CI/CD。
- 開始導入跨平台的底層程式碼。

## 未來的目標
BU的簡單需求都可以省力的達成，可以花更多人力在長期目標上，開始討論如何為BU節省開發成本，討論規劃出一些驗證工具讓BU或客戶可以更簡易找到問題，並導入自動化測試流程，在原始程式碼 commit 時 GitHub 就可以執行對應的腳本測試常用的 hub chip，讓開發功能時同時也能確認在大部分的 hub chip 底下運作是正常的：

- 三大平台 (Linux、Mac、Windows) 底層程式碼共用。
- 導入自動化測試。
- 增加 Diagnostic Tool 讓開發驗證可以更快速找到問題點。

[Secure Firmware Recovery](general/secure-firmware-recovery.html)
本文件詳細介紹了 OCP 安全韌體恢復 (Secure Firmware Recovery) 的規範，包括系統架構、恢復觸發條件與場景、恢復流程、安全威脅模型等。這與本案例的軟體建構之路有以下關係：

1. 在第三步的大躍進中，提到導入 CI/CD 流程，這與 Secure Firmware Recovery 的自動化恢復機制有關，可以確保即使韌體出現問題，也能透過低階通道進行恢復。
2. 未來目標中提到導入自動化測試，這也與 Secure Firmware Recovery 的偵測機制相關，能夠及時發現韌體的完整性問題。
3. 整體而言，Secure Firmware Recovery 規範提供了一個可靠的韌體恢復框架，有助於解決軟體建構過程中出現的韌體相關問題。

[2025 Minimum Elements for a Software Bill of Materials (SBOM)](general/2025-minimum-elements-for-a-software-bill-of-materials-sbom.html)
本文件介紹了 2025 年 SBOM 的最低要素，這與本案例的軟體建構之路有以下關係：

1. 在第三步的大躍進中，提到從 SVN 遷移到 GitHub，這與 SBOM 的自動化生成和機器可讀性要求相符。
2. 未來目標中提到導入自動化測試，這也與 SBOM 的自動化支援需求相關，能夠更好地管理軟體組件的依賴關係和風險。
3. 整體而言，SBOM 的最低要素為軟體供應鏈的透明化和風險管理提供了標準化的基礎，有助於解決本案例中軟體建構過程中的混亂和複雜性問題。

總的來說，本案例的軟體建構之路與 Secure Firmware Recovery 和 SBOM 兩個規範都有密切關聯，體現了從混亂到有序的轉變過程，以及如何通過標準化和自動化手段來提升軟體建構的可靠性和安全性。
