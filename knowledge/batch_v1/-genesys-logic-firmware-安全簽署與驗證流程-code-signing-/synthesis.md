報告書：Genesys Logic Firmware 安全簽署與驗證流程 (Code Signing)

## 💡 簡介 (Introduction)

Genesys Logic 為了滿足 HP Code Signing 的安全要求，設計了一套完整的韌體簽署與驗證架構。此架構的核心目標是確保所有簽署操作都符合 FIPS 140-2 Level 3 的安全標準，並採用 ECDSA nistp256 加密演算法。在滿足嚴格安全合規性的前提下，Genesys Logic 還需要建立一套兼具成本效益、可擴展性與災難恢復能力的內部簽署系統。

## 🔑 Code Signing 資源與實作方案 (含 FIPS 合規要求)

Genesys Logic 選擇採用「經 FIPS L3 認證的 USB eToken 硬體，並搭配自行開發的 SDK 應用程式」作為最終方案 [`3rd-party-code-signing-specification-ecdsa`]。此方案將金鑰安全地儲存在防竄改硬體中，同時透過自研的金鑰管理系統（KMS）實現高度客製化的簽署流程與嚴格的稽核管控。

## 🏛️ eToken 簽署系統架構解析

Genesys Logic 的 eToken 簽署系統架構包含以下主要元件：

1. FIPS L3 認證的 USB eToken 硬體設備，用於安全地儲存私密金鑰。
2. 自行開發的 SDK 應用程式，提供簽署、驗證等功能。
3. 自研的金鑰管理系統（KMS），實現金鑰的生成、佈署、撤銷等全生命週期管理。

此架構確保了金鑰的安全性，同時也提供了高度客製化的簽署流程與嚴格的稽核管控。

## 🔑 Genkey 金鑰生成與佈署流程解析

Genesys Logic 的金鑰生成與佈署流程如下：

1. 使用 FIPS L3 認證的 eToken 硬體生成 ECDSA nistp256 金鑰對。
2. 將公鑰部署到簽署系統中，並妥善保管私鑰。
3. 透過自研的 KMS 系統管理金鑰的生命週期，包括金鑰的撤銷、更換等操作。

整個流程確保了金鑰的安全性和可控性。

## 📜 Sign File 簽署檔案工作流程

Genesys Logic 的簽署檔案工作流程如下：

1. 使用 SDK 應用程式連接 eToken 硬體設備。
2. 從 KMS 系統獲取私鑰，並使用 ECDSA nistp256 演算法對目標檔案進行數字簽名。
3. 將簽名結果與原始檔案合併，形成最終的已簽名二進位檔案。

此流程確保了簽署過程的安全性和可審計性。

## 🔄 端到端的程式碼簽署與驗證完整流程

Genesys Logic 的端到端程式碼簽署與驗證流程如下：

1. 開發人員將程式碼提交至版本控制系統。
2. 持續整合（CI）系統自動觸發簽署流程，使用 eToken 硬體和 SDK 應用程式對程式碼進行數字簽名。
3. 簽名後的二進位檔案被部署到目標系統。
4. 目標系統在啟動時，會使用內置的驗證機制對二進位檔案進行合法性驗證 [`3rd-party-code-signing-specification-ecdsa`]。

整個流程確保了程式碼的完整性和可信賴性。

## 🚀 程式碼簽署與驗證流程 (Boot Code Verify)

Genesys Logic 的程式碼簽署與驗證流程包括以下步驟 [`code-sign-ecc-key`]：

1. 使用 ECDSA nistp256 演算法生成公私鑰對。
2. 使用私鑰對目標程式碼進行數字簽名。
3. 將簽名結果與原始程式碼合併，形成最終的已簽名二進位檔案。
4. 在目標系統啟動時，使用預先部署的公鑰對二進位檔案進行合法性驗證。

此流程確保了啟動程式碼的完整性和可信賴性。

## eToken 集中式管理與安全存取服務平台(未來架構)

為了進一步提升 eToken 簽署系統的可擴展性和災難恢復能力，Genesys Logic 正在規劃建立一個集中式的 eToken 管理與安全存取服務平台。此平台將提供以下功能：

1. 集中管理多個 eToken 設備，實現統一的金鑰生命週期管理。
2. 提供安全的遠程存取服務，使開發人員能夠安全地連接 eToken 設備進行簽署操作。
3. 支援高可用性和災難恢復機制，確保簽署服務的持續運行。

此未來架構將進一步提升 Genesys Logic 的 Code Signing 解決方案的可靠性和擴展性。
