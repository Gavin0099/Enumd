### 普通的做法
- Code Review
- 別的部門提出需求時，要有正式的SPEC，並且要有review動作，而且在完成時，整理成一份完整的文件以便留存
- 利用相關軟體，讓大家了解目前BU 需求的優先順序
- 每個人分享自己擅長的技術, 越淺顯易懂越好, 不見得與工作會有相關性。這樣可以增加大家的知識, 也可以了解不同部門的同事在做的東西, 或許分享的一些技術未來會用在工作上也不一定。例如軟體組可以分享不同的程式語言, 做些基本的介紹或語法
- 持續精進做事，先了解自己的作法，定期詢問他人或是自我檢視何處可以進步。提出問題，對他人的作法有疑慮，或是無法清楚該如何動作，主動提出\告知。該如何做?何時做?
- 關於Hub&PD相關工作，規劃Wiki 資訊，能讓新人比較容易上手，且能找到想要的資訊。
- 開發的重要專案、軟體都需做個詳細手冊，從概念到實作進行說明，可幫助自己事後回憶、或讓要加入專案的人更有效率地上手。
### 天馬行空的做法
- 3─2─2
「3─2─2」代表什麼？發想人之一的哈佛商學院學者惠蘭斯（Ashley Whillans）解釋，「3」是3天實際到班，「2」是兩天在家工作，最後的「2」則是原本的例休。
對台灣來說，混合工作的出發點是：對主管來說，不能再局限於「視覺管理」（Seeing Management），看得到員工才放心，必須強化對員工的信任感，不管透過什麼方式，只要員工能自主完成分內工作，那就不必硬性要求員工非得進辦公室。
- 目標與關鍵成果（OKR）
Objectives & Key Results，是透過每一組目標（objectives）搭配2～4 個關鍵成果（key results），讓團隊了解「要做什麼」及「如何做」。
