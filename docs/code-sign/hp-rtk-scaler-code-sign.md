# 如何判斷SW或是HW Code Sign
目前是用Scaler FW的檔名, 配合INI的設定, 來決定走HW還是SW的流程, 以下為INI的內容
[Scaler]
RtkSupportModel=E24m,E27m,E34m
E24m=RTD2778
E27m=RTD2788
E34m=RTD2798
CodeSign Type 0:None, 1: HW, 2:SW
[RTD2778]
NewDifferBank=1
CodeSign=2
HwWpGpio=0x102a
HwWpGpioReserved=0xf8
HwWpGpioValue=0x01
HwWpPin=0xffe2
HwWpPinReserved=0xfb
HwWpPinValue=0x04
[RTD2798]
NewDifferBank=1
CodeSign=1
HwWpGpio=0x1064
HwWpGpioReserved=0xf8
HwWpGpioValue=0x01
HwWpPin=0xfe54
HwWpPinReserved=0xfe
HwWpPinValue=0x01
不同的Scaler Type會有不同的Code Sign方式, 解除Write Protect的參數也不相同, 因此這些資訊必須事先定義在INI內
RtkSupportModel=E24m,E27m,E34m
程式會根據Scaler Bin檔檔名, 去抓對應的設定
例如Scaler Bin為E24M_G4_M238HCA_LCZ_C2_715GC248_M0A_BIT001_V0.0.0.5_20210426_DigitalSign_User.bin
程式會抓到檔名開頭的E24M, 並且在INI內找到對應的Scaler Type
E24m=RTD2778
程式再去[RTD2778]內將對應的設定全部讀入 (下圖紅框, CodeSign=2, 就會走SW驗證的流程)

# 程式INI說明
上述的例子, 只適用於HP ISP Tool, 因為HP ISP Tool不會知道使用者選了哪一個Scaler的FW

# Bin檔說明
簡單來說, SW驗證的簽章與bin檔不同檔案, HW驗證的簽章與bin檔同一個檔案
SW驗證: 會有兩個檔案, 檔名相同只有副檔名不同, 一個是.bin, 一個是.bin.sig
.bin是正常的Bin檔不包含簽章, .bin.sig為簽章內容, 以E24m為例有以下兩個檔案
E24M_G4_M238HCA_LCZ_C2_715GC248_M0A_BIT001_V0.0.0.5_20210426_DigitalSign_User.bin
E24M_G4_M238HCA_LCZ_C2_715GC248_M0A_BIT001_V0.0.0.5_20210426_DigitalSign_User.bin.sig
HW驗證: 只會有一個檔案, 包含了簽章在內, 副檔名是.bin.sig, 例如E34m的bin檔為E34M_TPM340Y1_2H012_21B_715GC248_M0A_XVT001_V0.0.0.5_20210426_DigitalSign_User.bin.sig
上圖為RTK文件的內容, 其中Bank就是Block, 一個Bank的大小定義是64KB, 因此一個RTK的Scaler Bin檔, 大小一定是64KB的倍數
如果需要做數位簽章驗證, 則會將相關的內容放在最後一個Bank, 例如檔案大小為1245184, 總共會有1245184 / 65536 = 19個bank, 則65536 X 18 = 1179648 = 0x120000開始會存放簽章相關資料
以下介紹每個欄位內容
Header: 目前拿到的FW, Header都是ZSTR這個字串, 但不確定RTK以後的案子會不會都是一樣, 需要再與RTK做確認
Version: 目前拿到的FW, 內容都是0, 還不確定有什麼作用
User Type: 目前拿到的FW, 內容不太一樣, 有的都是0, 還不確定有什麼作用
Key Length: Key的長度, 算法為KeyBuf[0] + (KeyBuf[1] << 8) + (KeyBuf[2] << 16) + (KeyBuf[3] <<24)
例如四個Byte依序為0x04,0x03,0x00,0x00, 則長度為0x304
Signature Length: Signature的長度, 算法與KeyLength相同, 如果為SW驗證, Signature Length = 0, 因為Signature會在另一個檔案 (SW驗證的簽章長度為0, 這一點RTK文件內並沒有提到)
Key Data: 接在Signature Length後面, 長度由Key Length取得
Signature Data: 接在Key Data後面, 只有HW驗證的檔案才會有signature
---
# SW Key
有兩種Key會從流程中得到
1. 從Flash得到的Key (以下簡稱Flash Key), 取得方式: 文件的4.13.7 Get Current Key Address, 先取得Key的位址後, 再用4.19.1 Get Current Verify Key from Flash的方法得到Key
1. 從Bin檔得到的Key (以下簡稱Bin Key), 取得方式: 文件的4.12 Extract FW/Key/Signature, 就是上面Bin檔的說明
Flash Key: 用RTK提供的DLL來驗證簽章 (.bin.sig的內容), 不用寫進Flash
Bin Key: 流程當中的Next Verify Key, 需要寫進Flash (4.18 Write Next Verify Key For New Differ Bank), 而寫到Flash的位址請參考 (4.13.6 Get Next Verify Key Address取得)
---
# HW Key
與SW比較, HW的流程相對簡單, Key和簽章都是從Bin檔讀出來, 然後寫到Flash某特定的位置後, 讓Scaler自己去做驗證即可
Key: 從Bin檔讀出來, 寫入flash的位置為4.13.6 Get Next Verify Key Address
簽章: 從Bin檔讀出來, 寫入flash的位置為4.13.7 Get Signature Address
---
# 燒錄流程說明
## HW驗證
燒錄流程最後兩個步驟分別是Reset MCU與New Differ Bank After ISP
下了Reset MCU之後會造成Hub斷電, 此時FW會去進行簽章的驗證, 而驗證的時間與Bin檔的大小相關, 
因此程式需要在Reset MCU後Delay夠久的時間(*1), 讓FW去做驗證. 而New Differ Bank After ISP就是下指令去取得驗證的結果, 驗證結果成功, 才是完成整個燒錄流程
New Differ Bank After ISP會失敗的原因
(*1)RTK提供的公式是 Bank數 X 1.5 + 3 = Delay秒數, 例如Bin檔有10個Bank, 就需Delay 10 X 1.5 + 3 = 18秒, 但後來經過實驗, 這個時間不夠, 應該要 Bank數 X 1.5 + 10 比較保險
