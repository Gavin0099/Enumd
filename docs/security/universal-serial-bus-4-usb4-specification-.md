## 📘 2. USB4 架構總覽（Architectural Overview）

### 🔹 USB4 基本概念

- USB4 是一種 **有線匯流排（cable bus）**，可支援主機與多種周邊裝置之間的資料交換。

- 不同於過去版本，USB4 **主機也能設定周邊裝置之間的資料交換通道**。

- 支援 **多設備同時操作（simultaneously accessible peripherals）**，包含熱插拔操作。

---

### 🔹 與先前版本的差異

- **取代 USB 3.2 功能**（透過 USB Type-C® 接口），同時仍保留 **USB 2.0 並行運作**。

- 繼承 USB 3.2 的 **Enhanced SuperSpeed USB 架構** 作為核心傳輸方式。

- 相較於 USB 3.2，USB4 採用 **連線導向（connection-oriented）、通道導向（tunneling）** 架構：

  ✅ 支援多種協定在同一實體介面上並行運作
  
  ✅ 總帶寬可動態分配於不同協定（例如 USB、DisplayPort、PCIe 等）
  
  ✅ **傳輸速率最高達 80 Gbps（Gen 4 x 2）**

---

### 🔹 通道技術（Tunneling Architecture）

USB4 引入「協定通道化」概念，允許下列協定並行運作：

| 協定類型 | 說明 |

| 🔹 USB3（Enhanced SuperSpeed） | USB 資料傳輸 |

| 🔹 DisplayPort（DP） | 顯示資料通道 |

| 🔹 PCI Express（PCIe） | 負載/儲存型資料通道 |

| 🔹 Host-to-Host | 主機間資料傳輸 |

- 各協定通道**獨立運作**，共用 USB4 的傳輸層與實體層。

- 亦支援 **匯流排管理封包與設定封包分配（例如流量管理、拓撲管理）**。

---

### 🔹 備註

- USB4 可透過單一實體介面，整合 **顯示、儲存、資料傳輸等多重用途**。

- 為高效能裝置互通與整合提供更大的彈性與效能擴展。

## 🔧 2.1 USB4 系統描述（USB4 System Description）

### 🧩 雙匯流排架構（Dual Bus Architecture）

- USB4 建構於 **USB 3.2 的雙匯流排架構基礎上擴充而成**。

- 支援 **向下相容（Backward Compatibility）**：

  - 最低從 USB 2.0 起，逐層向上相容到 USB 3.2，再到 USB4。
  
  - 根據 **互連元件支援的最高匯流排等級** 自動協調運作。

---

### 🔗 協定通道（Protocol Tunneling）與適配器（Protocol Adapter）

- 各協定通道需透過對應的 **Protocol Adapter** 進行連結。

- 處理方式依協定而異：

| 協定類型 | 適配邏輯需求 | 處理方式 |

| USB3 GenX、PCIe | ✅ 需要 | 透過原生 USB Hub、PCIe Switch 來進行封包路由與緩衝處理 |

| DisplayPort、Host-to-Host、USB3 GenT | ❌ 不需要 | 直接建立端到端通道（End-to-End Tunnel），無需中介裝置 |

---

### ⏱️ 時間同步（Time Synchronization）

- USB4 架構中透過 **分散式時間管理單元（TMUs, Time Management Units）** 來實現時間同步。

- 每個 Router（路由器）模組都配置 TMU，用以在整個 USB4 Fabric 內保持 **高精度同步**。

| USB3 UP Adapter | 與 Host 或 Hub 建立 USB3 連線 |

| PCIe UP Adapter | 提供 PCIe 資料上傳能力 |

| DP OUT Adapter | 提供顯示輸出功能 |

| Enhanced SuperSpeed / USB 2.0 Function | 與舊匯流排兼容的裝置功能 |

### 🔄 匯流排互動（總結）

| 匯流排類型 | 功能與走向 |

| USB 2.0 Bus | 並行獨立，支援最基本裝置連線 |

| USB3.x (Enhanced SuperSpeed) | 傳統高速資料傳輸，仍廣泛使用 |

| USB4 Fabric | 整合 USB/PCIe/DP 等協定，提供多任務通道 |

- 所有模組都具備 **TMU 時間同步模組**，確保在高速與多協定運作下的資料一致性。

- 每個模組皆可依需求 **進行協定轉譯（Protocol Adapter）**，從 USB 3.x、PCIe 到 DP 皆支援。

## 🧱 2.1.1 Architectural Constructs（架構基本構造）

USB4 架構由多個核心構造（Construct）所組成，其中最關鍵的是 **Router（路由器）**。

---

### 🔹 2.1.1.1 Routers（路由器）

### ✅ 功能定義

- **Router 是 USB4 架構中的基礎組件**，負責：

  - 將「通道化協定（Tunneled Protocol）」流量映射為 USB4 封包
  
  - 在 USB4 Fabric 中進行封包路由（Routing）
  
  - **透過 TMU（Time Management Unit）進行全系統時間同步**

### 🔄 與 Hub 的區別

- Router ≠ USB4 Hub

  - Router 是協定與封包的核心交換邏輯（flat, point-to-point, configurable switch）
  
  - Hub 是更高層次的裝置集合概念，內部通常會**包含 Device Router**

### 🧭 發現與設定

- Router 的發現與初始化由 **USB4 Host 內部的 Connection Manager** 負責。

### 🧩 典型配置

- 一個 Router 通常對應於以下任一裝置：

  - **USB4 Host**
  
  - **USB4 Hub**
  
  - **USB4 Device**

### 📦 Router 類型

| 類型 | 說明 |

| **Host Router** | 位於 USB4 主機端，管理協定通道與封包流向 |

| **Device Router** | 位於裝置端或 Hub 中，負責封包終端處理與協定解封裝 |

## 🔌 2.1.1.2 Adapters（轉接器）

### 🧩 基本說明

- 每個 USB4 Router 最多可包含 **64 個 Adapters**。

- Adapter 是 Router 與外部世界之間的介面，負責協定轉換與資料傳遞。

- Adapters 分為 **三大類型**：

---

### 🔹 1. Protocol Adapter（協定轉接器）

> 用於在「原生協定」與「USB4 通道（Tunnel）」之間進行轉換。

### ✅ 支援的協定類型：

| 協定類型 | 功能說明 |

| **USB3 Adapter** | 將 Enhanced SuperSpeed 封包導入/導出 USB4 |

| **DisplayPort (DP) Adapter** | 將影像資料進行 DP 通道封裝 |

| **PCIe Adapter** | 提供 PCI Express 通道轉換 |

| **Host Interface (HI) Adapter** | 對接主機內部介面（如 SoC） |

---

### 🔹 2. Lane Adapter（通道轉接器）

> Lane Adapter 為物理通道（Lane）提供連接介面。

- 一個 USB4 Port 中的每一個 Lane 都會對應一個 Lane Adapter。

- 負責物理層的資料進出控制與對接。

---

### 🔹 3. Control Adapter（控制轉接器）

> 一種邏輯轉接器（非實體介面），專責控制封包處理。

- 每個 Router 僅有 **一個 Control Adapter**。

- 功能如下：

  - 處理發送給該 Router 的控制封包（如設定與管理）
  
  - **產生控制封包**，傳送至 USB4 Host 端的 Connection Manager

---

### ✅ 補充備註

- Protocol Adapter 為 USB4 多協定通道運作的關鍵，讓 USB、DP、PCIe 可在同一匯流排上並行。

- Control Adapter 雖為邏輯元件，但在裝置管理與拓撲建立中扮演極重要角色。

## 🔗 2.1.1.3 USB4 Ports and Links（埠與連結）

### 🧩 USB4 Port 定義

- **USB4 Port 是 USB4 連結（Link）兩端的功能實體**。

- 每個 Port 包含以下組件：

  - **Tx（Transmit）與 Rx（Receive）Lanes**：USB4 的主要數據傳輸通道
  
  - **Sideband (SB) 通道**：兩線式通道，包括 `SBTX` 與 `SBRX`，負責輔助控制訊號與管理資料

---

### 🔗 USB4 Link 類型

| Link 類型 | 使用的 Lane | 傳輸特性 | 限制條件 |

| **Single-Lane Link** | 僅使用 Lane 0（Lane 1 被禁用） | 單通道傳輸 | 僅限 **Gen 2 / Gen 3** |

| **Aggregated Link** | 同時使用 Lane 0 與 Lane 1，邏輯上合併為單一通道 | 雙 Lane 聚合，可提供更高帶寬 | 可運作於任一速率級別（Gen 1～Gen 4） |

> ✅ Aggregated Link 可大幅提升頻寬（如 40 Gbps 或 80 Gbps）

---

### 🎯 小結與應用重點

- USB4 Port 是資料交換的端點，負責建立高頻寬數據通道。

- 根據裝置能力與連線品質，USB4 可動態選擇使用單一或聚合通道。

- **Sideband 通道（SBTX/SBRX）是 Link 中除數據外關鍵控制通訊機制。**

## 🔗 Aggregated USB4 Link 說明

### 🔸 聚合通道（Aggregated Link）

- 由 **兩個實體 Lane（Lane 0 + Lane 1）組成**。

- 提供更高的資料吞吐量（如 40 Gbps 或 80 Gbps）。

- 所有 USB4 裝置都需支援至少 Single-Lane，但高階裝置可進行 Aggregation。

---

### 🧱 Link 對稱性類型

| 類型 | 說明 | 適用範圍 |

| **Symmetric Link** | 雙方使用**相同數量的傳送器與接收器**（如：2 Tx, 2 Rx） | Gen 2、Gen 3、Gen 4 |

| **Asymmetric Link** | 一端有 **3 Tx / 1 Rx**，另一端為 **1 Tx / 3 Rx**（不對稱傳輸能力） | 僅限 **Gen 4** 支援 |

> ✅ Gen 2 & Gen 3 僅支援 Symmetric 模式

✅ Gen 4 可進行傳輸資源不對稱配置，提高效率與資源分配彈性

---

### 🧩 Sideband Channel（SBTX / SBRX）

- 雙方仍透過 Sideband 通道進行控制資訊與狀態交換。

- 即使數據通道為聚合，**控制面仍獨立處理**。

---

### 🎯 小結與應用建議

- Aggregated Link 是 USB4 的**高效能傳輸主力模式**，支援多協定並行（USB3、DP、PCIe）。

- 若需高頻寬、低延遲通訊場景（如顯示器、儲存擴充），應優先選擇支援 Aggregated 的裝置。

- 連結對稱性設定由 **Connection Manager 自動協商決定**。

## 🔗 USB4 Link 與 Fabric 通訊關係

### 📦 Link 的作用

- **USB4 Link 是 USB4 Fabric 的主通道**，負責傳遞兩端 Router 間的封包資料。

- Link 支援以下兩類封包：

  - **Tunneled Protocol 封包**（如 USB3、DP、PCIe）
  
  - **管理封包**（bus management）

---

### 🧩 Link 組成要素

| 組件 | 功能 |

| **USB4 Port** | 提供傳輸功能（Tx/Rx），為每個 Link 的端點 |

| **Sideband Channel** | 協助初始化與管理 Link，包含 `SBTX/SBRX` |

| **USB 2.0 Bus** | 與 USB4 並行傳輸，確保兼容性 |

| **Configuration Channel (CC)** | USB Type-C 的設定與角色協商控制通道 |

| **電源相關線路** | `VBUS`、`VCONN` 與 `GND` 提供電源供應與輔助控制 |

---

- 在實作 USB4 Type-C 裝置時，需考量以下元件整合：

  - **USB4 Port（含 Link 層）**
  
  - **USB 2.0 傳輸路徑**
  
  - **Sideband & Configuration Channel**
  
  - **電源管理線路與邏輯**

## 🧩 2.1.1.4 USB4 Devices（USB4 裝置）

### 🔹 基本定義

- **USB4 Peripheral Device（USB4 周邊裝置）** 是連接至 USB4 Host 的終端裝置。

- 它**僅具備一個上行埠（Upstream Facing Port, UFP）**，**不具備下行埠（DFP）**。

---

### 🧠 組成元件

| 元件 | 功能說明 |

| **Device Router** | USB4 傳輸的核心交換控制邏輯 |

| **Enhanced SuperSpeed Function** | 支援 USB 3.x 資料傳輸（非必要） |

| **USB 2.0 Function** | 提供向下相容的基本 USB 功能（非必要） |

| **PCIe Function** | 支援 PCI Express 協定封包傳輸（非必要） |

| **DisplayPort Source 或 Sink** | 可作為視訊輸出（Source）或輸入（Sink）（非必要） |

> ✅ 這些功能模組是 可選擇性的（Optional），視裝置設計與應用場景而定。

---

### ⚙️ 支援的 USB4 傳輸速度等級

| 傳輸模式 | 速率 | 是否必須支援 |

| **Gen 2** | 20 Gbps | ✅ 必須支援 |

| **Gen 3** | 40 Gbps | 🔘 選擇性支援 |

| **Gen 4** | 80 Gbps | 🔘 選擇性支援 |

> 📌 所有 USB4 裝置至少需支援 Gen 2，其他速率則依產品定位與實作而定。

---

### ✅ 小結

- USB4 Device 是單一上行介面的終端裝置，支援多種協定通道整合（USB3、DP、PCIe）。

- Device Router 為核心結構，配合各協定功能模組進行對應封包處理與轉換。

- 適用於高效能、可視訊傳輸或高速儲存的裝置設計（如顯示器、外接顯卡、外接硬碟）。

## 🧩 2.1.1.4.2 USB4 Hub（USB4 集線器）

### 🔹 基本定義

- USB4 Hub 是一種 **具備多個 USB4 下行埠（DFPs）與單一上行埠（UFP）** 的裝置。

- 功能類似於 USB 3.2 Hub，提供 **埠擴充（Port Expansion）** 能力。

- 結構上採 **樹狀拓撲**，由上行埠連接至 Host，再分配至多個下行埠。

---

### 🧠 組成元件

| 元件 | 說明 |

| **Device Router** | USB4 的核心交換路由邏輯 |

| **Enhanced SuperSpeed USB Hub** | 支援 USB 3.x 裝置的擴充能力 |

| **PCIe Switch** | 實作 PCIe 通道擴充 |

| **USB 2.0 Hub** | 提供對舊版 USB 裝置的相容性支援 |

---

### 🔁 埠支援與相容性

- **Upstream Facing Port（UFP）**：僅一個

- **Downstream Facing Ports（DFPs）**：一個或多個

- **每個下行埠都需支援**：

  - ✅ USB4 裝置連接
  
  - ✅ 向下相容的 USB 3.2 裝置
  
  - ✅ 向下相容的 USB 2.0 裝置

> 📎 USB4 Hub 必須能根據接入裝置自動對應通訊協定（USB4 / USB3 / USB2）

---

### 📺 DisplayPort Alt Mode 要求

| 項目 | 說明 |

| **支援要求** | USB4 Hub **所有下行埠必須支援 DisplayPort Alt Mode** |

| **規範依據** | 詳細要求請參考《USB Type-C Specification》 |

---

### ⚙️ 傳輸速率支援

| USB4 規格等級 | 是否支援 |

| **Gen 2（20 Gbps）** | ✅ 必須支援 |

| **Gen 3（40 Gbps）** | ✅ 必須支援 |

| **Gen 4（80 Gbps）** | 🔘 選擇性支援 |

---

### ✅ 小結

- USB4 Hub 是為 **多埠擴充而設計** 的核心設備，提供多協定支援與高度相容性。

- 與 USB4 Device 不同，Hub 同時扮演路由器、轉接器與協定分流的多重角色。

- 它不僅是 USB Hub，更是 **DisplayPort、PCIe、USB3、USB2 多通道的通訊節點**。

## 🧩 2.1.1.5 USB4 Host（USB4 主機）

### 🔹 基本定義

- USB4 Host 是整個 USB4 拓撲架構的 **起點（來源）裝置**，負責管理與初始化所有連線與協定交換。

- 為了支援多協定傳輸與兼容性，USB4 Host 結合多個模組組成。

---

### 🧠 組成元件

| 元件 | 功能說明 |

| **Host Router** | 中樞路由器，負責協定封包轉換與路由（含 TMU） |

| **USB 2.0 Host** | 支援向下相容的 USB 2.0 裝置 |

| **Enhanced SuperSpeed Host** | 支援 USB 3.x 裝置傳輸 |

| **DisplayPort Source** | 提供 DP 音視頻輸出能力 |

| **PCIe Controller / PCIe Switch** | （選配）支援與裝置之間的 PCIe 通訊 |

---

### ⚙️ 支援的 USB4 傳輸速率

| USB4 規格等級 | 是否支援 |

| **Gen 2（20 Gbps）** | ✅ 必須支援 |

| **Gen 3（40 Gbps）** | 🔘 選擇性支援 |

| **Gen 4（80 Gbps）** | 🔘 選擇性支援 |

---

### 📺 DisplayPort Alt Mode 要求

| 項目 | 說明 |

| **必備功能** | 所有 Downstream Facing Ports 必須支援 DisplayPort Alt Mode |

| **依據** | 詳細定義與規範請參考《USB Type-C Specification》 |

---

### 🎯 小結

- USB4 Host 是整個 Fabric 的主控節點，啟動連線協議、分配通道並維持時間同步。

- 它需具備 **多協定整合能力**，並與各類周邊裝置（USB、PCIe、DP）協調資料傳輸。

- 若目標應用包含高頻寬需求（如 eGPU、8K 顯示、多重儲存通道），建議實作 Gen 3 或 Gen 4 支援。

## 🔄 2.1.1.6 Re-timers（重定時器）

### 📌 定義

- **USB4 Re-timer** 是專門用於改善高速傳輸中訊號完整性的物理層元件。

- 每個 USB4 Port 最多可搭配 **2 顆 On-Board Re-timers**。

- 相關規格詳見《USB4 Re-Timer Specification》。

### 📦 功能

- **重建 Tx/Rx 訊號眼圖**

- **延伸連接距離**（特別是 USB4 Gen 3 / Gen 4 的 40Gbps/80Gbps）

- **減少 jitter / 遮蔽干擾**

---

## 🧠 2.1.1.7 Connection Manager（連線管理器）

### 📌 定義

- **Connection Manager** 是 USB4 Host 系統中的軟體邏輯單元。

- 它是整個 USB4 網域（Domain）內的控制中樞。

### ⚙️ 功能職責

| 功能項目 | 說明 |

| **設備列舉（enumeration）** | 發現並辨識連接的 USB4 裝置 |

| **路徑配置（Path setup/teardown）** | 建立或拆除協定通道與傳輸路徑 |

| **熱插拔處理（Hot Plug/Unplug）** | 動態新增/移除裝置時保持系統穩定 |

| **頻寬管理（Bandwidth management）** | 根據需求分配 USB4 Fabric 的傳輸資源 |

> ✅ Connection Manager 僅存在於 USB4 Host 裝置中

---

## 🧰 2.1.2 USB4 Mechanical（USB4 機構設計）

### 📘 規格來源

- 所有 USB4 的連接器與線材機構細節，皆依照《USB Type-C Specification》。

### 🔌 上/下行連接說明

| 裝置類型 | 上行埠（UFP） | 下行埠（DFP） |

| **USB4 Peripheral Device** | ✅ 有 | ❌ 無 |

| **USB4 Hub** | ✅ 有 | ✅ 有（1 或多個） |

| **USB4 Host** | ❌ 無 | ✅ 有（1 或多個） |

- **USB Type-C 接頭本身**可透過 PD 和 CC 設定來決定埠的角色（上行或下行），具備**可逆性與角色切換能力**。

---

### 📝 小結

- USB4 架構中不只有資料傳輸層（Router/Adapter），也包含：

  - **實體層補償元件**（Re-timer）
  
  - **連線與協定管理邏輯**（Connection Manager）
  
  - **硬體接頭與方向協定**（Mechanical via Type-C）

這些設計確保 USB4 在高頻寬、多協定、多角色的複雜環境中，依然可以**穩定、動態與向下相容**。

## 🔋 2.1.3 USB4 Power（電源管理）

### 📌 核心觀念

- **USB4 本身不定義獨立的 VBUS 電源模型**。

- 所有電源管理（VBUS、VCONN、功率協商）皆依照以下規範執行：

  - 《USB Type-C Specification》
  
  - 《USB Power Delivery (PD) Specification》

> ✅ 也就是說，電源輸出/輸入能力（如 100W PD）完全由 Type-C + PD 控制，USB4 專注於資料通道。

---

## ⚙️ 2.1.4 USB4 System Configuration（系統配置）

### 📌 核心流程

- USB4 的連線建立與角色分配，遵循《USB Type-C Specification》的設定機制。

- 一旦連線完成，**USB4 Host 使用軟體層的 Connection Manager** 進行進一步配置與管理：

  - 設定協定通道（如 USB3、DP、PCIe）
  
  - 管理 Lane Bonding（聚合）
  
  - 管理熱插拔與頻寬調整

> ✅ 這意味著 USB4 的實際拓撲與通道配置是由 Host 控制，不是由裝置自行決定。

---

## ⚡ 2.1.5 Thunderbolt™ 3（TBT3）Compatibility Support

### 📌 相容性支援規範

| 裝置類型 | 是否必須支援 TBT3 相容性 |

| **USB4 Host** | 🔘 選擇性支援 |

| **USB4 Peripheral Device** | 🔘 選擇性支援 |

| **USB4 Hub** | ✅ 所有 DFP **必須支援 TBT3 相容性** |

| **USB4 Dock** | ✅ 所有 DFP **必須支援**🔘 UFP 可選擇是否支援 |

### 📦 實作模式

- 當與 Thunderbolt 3 裝置連線時，系統會自動進入 **Thunderbolt Alt Mode**。

- 模式協商流程依《USB Type-C Specification》執行。

> 📘 詳細支援 Thunderbolt 3 的 Router 實作要求，請參考 USB4 規範 第 13 章。

---

## 📝 小結

- **USB4 不再設計獨立的電源架構**，完全交由 Type-C + PD 控制。

- **所有裝置角色與連線協商，統一透過 Type-C 規範與 Connection Manager 管理**。

- **Thunderbolt 3 相容性是 USB4 標準的一部分，尤其是對 USB4 Hub 裝置而言是必備功能**，確保與現有 TBT3 裝置的生態整合。

## 🧩 2.1.6 USB Type-C Alternate Mode Compatibility Support（Alternate Mode 相容性）

### 📌 定義

- USB4 產品**可選擇性支援**《USB Type-C Specification》中所定義的 **Alternate Modes（替代模式）**。

- 例如：

  - **DisplayPort Alt Mode**
  
  - **HDMI Alt Mode**（較少見）
  
  - **Thunderbolt Alt Mode**（與 TBT3 兼容性緊密關聯）

> ✅ 是否支援哪種替代模式，由產品設計決定，非 USB4 強制要求。

---

## 🧠 2.2 USB4 Fabric Architecture（USB4 資料骨幹架構）

USB4 Fabric 是一種設計來同時支援多協定傳輸的底層資料架構（Transport Layer），具備高度可擴展性與動態管理能力。

---

### 🚀 支援速率（Signaling Rates）

| 規格代號 | 實際速率 | 備註 |

| **Gen 2** | 10 Gbps | 基礎傳輸層 |

| **Gen 3** | 20 Gbps | USB4 標準建議支援 |

| **Gen 4** | 40 Gbps | 高階裝置使用 |

| **TBT3 相容速率** | 10.3125 / 20.625 Gbps | 為與 Thunderbolt 3 相容而設計的速率調整選項 |

---

### 📦 核心特性一覽

| 特性 | 說明 |

| **Hop-by-hop credit-based flow control** | 每個節點之間使用「信用點數」來控管資料傳送，確保不會溢出 |

| **Bandwidth management & prioritization** | 支援通道分配與優先順序控制，可滿足多協定同時傳輸 |

| **透明的 Programming Model** | 由 Connection Manager 控制拓撲初始化與維護，對於協定通道與上層軟體是透明的 |

| **時間同步協定** | 跨多個 USB4 Domain 同步時鐘（由 TMU 單元執行） |

| **錯誤偵測與修復機制** | 含 CRC 校驗、重傳與封包修復能力 |

| **Link-level Power Management** | 對各個 USB4 Link 進行獨立的低功耗狀態控制（非全域 suspend） |

---

### ✅ 小結

- **USB Type-C Alt Mode 相容性**賦予 USB4 更大的彈性，支援影音等特殊應用模式。

- **USB4 Fabric** 是整個系統的傳輸骨幹，具備類似「網狀交換架構」的智慧通道調度能力，支援高頻寬與多協定混合使用。

### 📊 圖示：Figure 2-7 – USB4 Functional Stack Layers

堆疊分為五大區塊，從底層電氣訊號到高層協定轉換，提供 USB4 支援多協定與高頻寬的結構基礎。

---

### 🧩 功能層解說

### 🔻 **1. Physical Layer（實體層）**

> USB4 的最低層，負責實際的訊號傳送與收發。

- **Electrical Layer（電氣層）**

  - 負責物理信號驅動：電壓、速率（10/20/40Gbps）、線路品質。
  
  - 包含 PHY、Re-timers、Equalizer 等。

- **Logical Layer（邏輯層）**

  - 負責封包的格式定義、CRC、錯誤檢查、分段與重組。
  
  - 是 Transport Layer 與實體訊號之間的中介。

---

### 🛣 **2. Transport Layer（傳輸層）**

> 管理封包從來源到目的地的 hop-by-hop 流動。

- 提供 **credit-based flow control**。

- 支援封包緩衝、重傳、路由與時間戳記。

- 屬於 USB4 Fabric 的關鍵層。

---

### 🔄 **3. Protocol Adapter Layer（協定轉接層）**

> USB4 多協定支援的核心，讓 USB、DP、PCIe 可透過 USB4 Tunnel 傳送。

- 負責將原生協定（如 USB3、DP、PCIe）**轉換成 USB4 封包格式**。

- 每種協定會對應一個專屬的 Protocol Adapter。

---

### ⚙️ **4. Configuration Layer（配置層）**

> 提供設備與連線的控制設定介面。

- 包含控制通道管理（Control Packets）

- 管理設備初始化、連線建立與頻寬配置

- 與 Connection Manager 密切配合

---

### 📝 小結

| 層級 | 功能定位 | 說明 |

| Protocol Adapter | 協定整合 | USB/DP/PCIe 封裝成 USB4 |

| Configuration | 控制配置 | 裝置初始化、通道設定 |

| Transport | 路由傳送 | 封包控制與交換管理 |

| Logical | 封包處理 | 分段、檢查、重組 |

| Electrical | 訊號層 | 真實訊號傳輸與電氣驅動 |

### 🔍 說明目的

此圖展示 USB4 架構中三種核心 Adapter：

- **Protocol Adapter**

- **Lane Adapter**

- **Control Adapter**

分別對應在哪些 Stack Layer 中實作與運作。

---

## 🔌 各 Adapter 與對應層級關係

| Adapter 類型 | 涉及層級 | 功能說明 |

| **Protocol Adapter** | Protocol Adapter Layer + Transport Layer | - 將原生協定（USB3 / DP / PCIe）轉為 USB4 Tunneled 封包- 管理協定相關 flow control / buffer / 封包 header |

| **Lane Adapter** | Transport Layer + Physical Layer（含 Logical + Electrical） | - 管理 USB4 Port 實體傳輸通道（Lane）- 處理 Tx/Rx、Lane Bonding、Lane Mapping |

| **Control Adapter** | Configuration Layer + Transport Layer | - 負責控制封包處理：裝置發現、通道配置、熱插拔反應- 與 Host 的 Connection Manager 配合完成設定協調 |

---

### 📘 實作備註（實體 vs 邏輯）

| Adapter | 實作方式 | 補充說明 |

| **Protocol Adapter** | ASIC（IP Block） | 高速資料路徑，常整合在 USB4 Controller 中 |

| **Lane Adapter** | PHY + 控制邏輯 | 包含 PHY、MUX、Retimer 控制等 |

| **Control Adapter** | MCU / FW + 硬體協同 | 軟體參與較多（如 Hotplug 回應與 Path teardown） |

---

## ✅ 小結

- 此圖直觀反映「Adapter 並非單純一層」：

  - **Protocol Adapter 橫跨協定與傳輸層**
  
  - **Control Adapter 負責管理與配置**
  
  - **Lane Adapter 深入實體層，對應實體通道與收發行為**

- 對硬體設計而言，這些分層對應可協助模組化設計與驗證

- 對軟體來說，Control Adapter 是 Host stack 和硬體交換資訊的關鍵節點

## ⚡ 2.2.1.1 Electrical Layer（電氣層）

### 📌 定義

- 負責 **USB4 Link 的電氣訊號層級特性**，即實際在電纜上傳輸的位元訊號。

### 📦 核心功能

| 特性 | 說明 |

| **Scrambling（擾碼）** | 將訊號打散，避免連續比特造成 EMI 增強訊號完整性 |

| **Encoding** | USB4 採用 128b/132b 編碼（源自 PCIe Gen 3+）以增加效率 |

| **Jitter 控制** | 控制訊號邊緣的抖動，確保位元正確解析 |

| **Voltage 規格** | 包含電壓擺幅、眼圖要求與差分訊號控制（例如：800 mVpp-d） |

---

## 🧠 2.2.1.2 Logical Layer（邏輯層）

### 📌 定義

- **Logical Layer 負責在兩個 Router 間建立並維護 USB4 Link**。

- 處於 **Electrical Layer 與 Transport Layer** 之間，視 Transport 資料為 byte stream。

### 📦 提供的服務

| 功能 | 說明 |

| **Link Establishment & Maintenance** | 負責協商速率、Lane bonding、同步等動作 |

| **Scalable Performance** | 支援單/雙 Lane、Gen2/Gen3/Gen4 等不同速率與傳輸寬度 |

| **Error Detection & Recovery** | 使用 CRC、FEC（Forward Error Correction）等技術進行錯誤修復 |

| **媒介支援** | 可透過：- 被動線材- 主動線材- Re-timer進行傳輸適配 |

| **Clock Compensation** | 支援時脈偏移補償與同步（與 Re-timer 協同） |

| **Scrambling & FEC** | 避免 DC 偏移與重建數據精度 |

| **Link Power Management** | 支援鏈路進入低功耗狀態（例如：L1-like idle 模式） |

---

### 🔁 Sideband Channel 支援功能

Logical Layer 並非獨立運作，其啟動與狀態由 Sideband Channel（SBTX/SBRX）進行協調：

| 功能 | 說明 |

| **參數設定** | 初始化時設定 Tx/Rx FFE、EQ、速率、Lane 配置等 |

| **Re-timer 溝通** | 協助配置中繼器（如 BW9906）進行 FFE 或電壓擺幅調整 |

| **正確上/下電流程控制** | 確保 USB4 Link 與 Re-timer 的上電、關閉與喚醒序列正確執行 |

---

## ✅ 小結

| 層級 | 核心目標 | 運作方式 |

| **Electrical Layer** | 高速實體位元傳輸 | 定義訊號完整性規格 |

| **Logical Layer** | Byte Stream 鏈路建立與維護 | 包含協議協商、錯誤修復與 FEC |

- **Sideband Channel** 為整個 Link 初始化與狀態管理的輔助控制通道，非資料通道但極為關鍵

## 🛣️ 2.2.1.3 Transport Layer（傳輸層）

### 📌 定義

- **核心交換層（Switching Core）**，負責在 USB4 Fabric 上路由與轉送各種封包。

### 📦 功能說明

| 功能 | 說明 |

| **Tunneled Packet + Control Packet 路由** | 可分辨封包類型並正確轉送 |

| **Packet Format 定義** | 定義封包欄位、Header、QoS 等結構 |

| **Flow Control** | 採用 credit-based flow control（流量控制、避免 overflow） |

| **Time Synchronization** | 與 TMU 配合，支援跨設備時間同步 |

| **Protocol Muxing** | 多協定共用一個 USB4 Transport（如 USB3+PCIe+DP）時需進行封包標記與識別，稱為 MUX |

> ✅ Transport Layer 是整個 USB4 Fabric 的中繼節點核心邏輯，並與 Protocol Adapter 和 Configuration Layer 協同運作。

---

## ⚙️ 2.2.1.4 Configuration Layer（配置層）

### 📌 定義

- 負責 **Router 的組態設定與控制封包處理**，是與 Connection Manager 對接的核心界面。

### 📦 功能說明

| 功能 | 說明 |

| **處理 Control Packets** | 解析來自 Connection Manager 的設定封包 |

| **Router Configuration Tasks** | 如 Port 狀態設定、Link 配置、Adapter Mapping |

| **Addressing Scheme** | 為每個 Router 分配唯一識別位址以便封包送達 |

| **可靠傳輸機制** | 確保 Control Packets 不遺失（可重送、校驗） |

> 📌 Control Packets 是 Configuration Layer 與上層軟體（Connection Manager）之間的橋梁

---

## 🔌 2.2.1.5 Protocol Adapter Layer（協定轉換層）

### 📌 定義

- 負責在 USB4 Transport 封包與原生協定封包（USB3、DP、PCIe）間進行映射與轉換。

### 📦 功能說明

| 功能 | 說明 |

| **Tun­neled Protocol → USB4 Packet** | 將 PCIe/DP/USB3 封包包裝成 USB4 封包 |

| **USB4 Packet → Tun­neled Protocol** | 將 USB4 封包解包為原生協定格式 |

| **MUX-aware** | 每個 Protocol Adapter 僅處理指定協定封包，根據 Mux 標記正確送入處理模組 |

| **適配類型分類** | 詳細在 §2.2.10 中說明：包含 USB3、DP、PCIe、Host Interface 等類型 |

---

## 📘 小結：Stack 功能關係總覽

| 層級 | 功能重點 | 對應硬體單元 |

| **Protocol Adapter Layer** | 協定封包映射與解封裝 | 協定 Tunnel IP |

| **Configuration Layer** | 控制封包處理與組態設定 | Control Adapter / MCU |

| **Transport Layer** | 封包路由與流控、Mux | USB4 Switch Fabric / Router IP |

## 🌐 2.2.2 USB4 Fabric Topology（USB4 網路拓撲）

### 📌 定義

- USB4 的 **Physical Topology** 是一個 **連通圖（Graph）**，由 USB4 Host、Hub、Device 互相連接而成。

- USB4 支援 **非固定樹狀結構**，甚至可能形成「環狀路徑（Loop）」。

---

### 🌲 Spanning Tree 概念

> 為了避免控制封包在拓撲中無限循環，USB4 Host 的 Connection Manager 會主動建立一棵「Spanning Tree」 來排除 Loop 結構。

### 🔧 功能：

| 功能 | 說明 |

| **移除物理迴圈（Loop）** | 將實際的網路拓撲轉換成一棵無環（Loop-free）的邏輯樹 |

| **控制封包僅走 Spanning Tree** | Control Packets 只能在 Spanning Tree 節點之間傳遞 |

| **資料封包可跨越非樹路徑** | Tunneled Protocol 封包不受限於 Spanning Tree，可使用全圖做高效路由 |

| **最多支援 6 層級** | Host Router 可與最遠的 Device Router 相隔 **最多 5 hops**，整體為 6 層級拓撲 |

## ✅ 小結

- USB4 Fabric 並不強制採用固定樹狀架構，允許彈性連接甚至產生 Loop。

- 為確保控制封包不產生路徑震盪與死循環，Host 使用 **Spanning Tree Algorithm（如類似 MST）** 進行路徑約束。

- **資料封包仍能充分利用整個網路圖**，具備高頻寬、高效率與彈性路由能力。

## 🔀 2.2.3 Paths（邏輯通道路徑）

### 📌 定義

- 在 USB4 Fabric 中，一條 **Path** 代表一個**方向性邏輯連線**，其構成為多個 Router 之間的封包路徑。

- Path 的兩端通常為：

  - 兩個 **Protocol Adapters**
  
  - 或是 Connection Manager ↔ Control Adapter

---

## 🔧 路徑類型

| 路徑類型 | 說明 | 範例 |

| **End-to-End Path** | 通過多個 USB4 Link 形成一條完整通路 | USB3 Gen T（端對端）DisplayPort TunnelingHost-to-Host Tunneling |

| **Link-Local Path** | 僅存在於單一 Link 上，不穿越多個 Router | USB3 Gen X（如 USB3 裝置僅通過一層 Hub）PCIe Tunneling |

---

## 📦 封包轉送機制：HopID

> HopID 是 USB4 Fabric 中封包識別的關鍵欄位。

- 每個封包都會攜帶 **HopID**

- 每個 Router 根據目前 Link 的 HopID 查表決定：

  - 下一個 HopID
  
  - 要送往哪個 **Egress Adapter**

> 🔄 HopID 可在每一條 Link 間變更，是類似 Label Switching 的概念（如 MPLS）

## 🔄 資料通訊流程範例

### ➤ **以 USB3 資料為例：**

1. **Native Protocol Traffic**

  - 如 USB3 資料進入 Protocol Adapter

1. **Tunneled Packet**

  - 封裝為 USB4 Transport 層格式

1. **Transport Layer**

  - 指派 HopID，進行 Router 間封包跳轉

1. **Physical Layer**

  - 傳輸為 bit stream，經過電氣層與邏輯層處理

1. **USB4 Link**

  - 穿越主動/被動線材，抵達下一跳

### 🧠 Control Packet vs Tunneled Packet 比較

| 類型 | 來源 | 功能 | 傳輸通道 |

| **Control Packet** | Connection Manager | 設定 Router、配置 Path、管理 HopID | 僅走 **Spanning Tree** |

| **Tunneled Packet** | Protocol Adapter | USB3/DP/PCIe 等協定資料封包 | 可走任意 Link（非限 Spanning Tree） |

---

## ✅ 小結

- USB4 並非單一層通訊協定，而是多層堆疊，**每層傳輸不同類型封包與控制資訊**

- Connection Manager 掌控 Control Packet 傳遞與 Router 組態

- Protocol Adapter 將原始協定以 Tunnel 封包格式傳輸於共享 USB4 Fabric

- Sideband Channel 則提供 Layer 外的初始化與電氣調整機制

## 📦 Control Packet 概念重點

| 項目 | 說明 |

| **功能目的** | 提供 Connection Manager 操控 Router（組態設定、建立 Path、同步） |

| **傳遞通道** | 僅透過 **Spanning Tree 上的 Link** 轉送 |

| **辨識方式** | 每個 Control Packet 包含一個 **Route String**，用來定位目標 Router |

| **路由邏輯** | 中間 Router 根據 Route String，將封包轉送至下一個 Port |

| **接收處理** | 目標 Router 的 **Control Adapter** 會解包並處理 Control Packet |

| **反向路徑** | Router 回報給 Connection Manager 時，也以 Route String 標記來源 |

### 🧠 Control Packet 在 Stack 中的流動（圖 (b)）

在每一個 Router 中，Control Packet 會經過：

1. **Electrical Layer**

1. **Logical Layer**

1. **Transport Layer**

1. **Configuration Layer**

  - 若為目標 Router，封包在這層被「Consume」
  
  - 若為中繼 Router，封包在這層被轉送（由 Control Adapter 執行轉發動作）

> 🔁 整個過程類似一個輕量級封包轉發機制，只針對少量控制指令，但極為關鍵

## ✅ 小結：Control Packet Routing 總覽

| 特性 | 說明 |

| **封包目標識別** | 使用 Route String 對應 Router 位置 |

| **傳遞通道限制** | 僅走 **Spanning Tree**（避免循環與不確定性） |

| **層級路由邏輯** | 依據各層功能層級分層處理 |

| **處理器角色** | Control Adapter 在各 Router 擔任轉發 / 消耗角色 |

| **來源 / 目標識別** | Route String 區分「誰送出」與「送到哪」 |

### 🔄 2.2.2.1.2 **Tunneled Packets（協定封裝封包）**

| 項目 | 說明 |

| **用途** | 用於攜帶封裝後的協定資料（如 USB3、PCIe、DP） |

| **封裝層** | 由 Protocol Adapter Layer 將原生協定轉為 USB4 格式 |

| **傳遞方式** | 經由 Transport Layer 根據 **Path（路徑）** 傳送 |

| **支援功能** | 支援多協定同時傳輸，並行多條 Tunnel |

| **特性** | 每個封包標記對應 HopID，支援跨多 Router 轉送 |

> ✅ Tunnel 需在資料傳輸前預先建立，配置路由與資源

---

### 🛠️ 2.2.2.1.3 **Link Management Packets（鏈路管理封包）**

| 封包類型 | 說明 | 傳輸範圍 |

| **Time Sync Packets** | 對時封包，協助 TMU 跨 Router 時鐘同步 | Link-local（單一 Link） |

| **Flow Control Packets** | 信用點控制，用於防止 Router buffer overflow | Link-local |

| **Idle Packets** | 當無資料傳輸時填補，維持 Logical Layer 的時序穩定 | Link-local |

> ✅ 這些封包皆僅存在於「單一 USB4 Link」中，由 Link 的兩端 Router 處理，不會穿越多層節點

---

### 🔗 2.2.2.1.4 **Ordered Sets（有序符號組）**

| 功能 | 說明 |

| **符號同步（Symbol Sync）** | 協助接收端對齊資料流中的位元與符號界限 |

| **Link Training** | 用於協商 Lane 的傳輸速率、FFE/EQ 參數等 |

| **De-skew** | 當多個 Lane 資料到達時間不同時，進行重組同步 |

- **位置層級**：Logical Layer

- **組成**：一組專屬符號組成（非資料封包），例如 TS1 / TS2 / SKP 等（類似 PCIe）

> ✅ Ordered Set 並非封包，而是 Symbol-level pattern，僅在邏輯初始化與 Lane 調整過程中出現

---

## ✅ 封包類型對照表（總結）

| 類型 | 對應層級 | 功能 | 跨 Router |

| **Tunneled Packet** | Protocol + Transport | 協定封裝資料 | ✅ 是 |

| **Control Packet** | Configuration + Transport | 組態設定 / 控制 | ✅ 是（Spanning Tree 限制） |

| **Link Management Packet** | Transport | Flow control / 時鐘同步 | ❌ 僅 Link local |

| **Ordered Set** | Logical Layer | Link training / De-skew | ❌ Symbol-only，非封包 |
